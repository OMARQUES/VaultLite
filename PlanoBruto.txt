Perfeito. Vou assumir **Codex como o agente de código da OpenAI**, usado via app/CLI/IDE, e montar o backlog no formato que ajuda um agente a trabalhar bem: tarefas pequenas, contexto explícito, plano antes da implementação em partes difíceis, e instruções persistidas em `AGENTS.md`, já que a própria documentação do Codex recomenda isso. A OpenAI também descreve o Codex como um agente que pode escrever código, revisar, corrigir bugs e trabalhar em sandbox isolado com o repositório carregado, o que favorece tickets autocontidos e verificáveis. ([OpenAI Developers][1])

# 1. Direção final do projeto

## Produto

Gerenciador de senhas **open source**, **self-hosted**, **zero-knowledge**, com foco em:

* cofre E2EE
* web app em **Vue 3 + TypeScript**
* extensão de navegador
* login items
* document items
* anexos criptografados
* gerador de senha
* sync
* organização básica
* import/export
* Account Kit
* device management
* troca de master password apenas com sessão autenticada

## Princípios inegociáveis

* o servidor **nunca** descriptografa o cofre
* o admin **nunca** recupera a master password
* sem master password e sem dispositivo já autenticado, **não há recuperação**
* o **Account Kit** ajuda no bootstrap de novo dispositivo, **não** no reset de senha
* o usuário define a própria senha no onboarding
* anexos são criptografados **no cliente antes do upload**

# 2. Como o backlog deve ser escrito pensando no Codex

Como a OpenAI recomenda “plan first” para tarefas difíceis e `AGENTS.md` para guidance reutilizável, o backlog deve evitar tickets vagos do tipo “fazer auth” e preferir tarefas com escopo pequeno, arquivos prováveis, restrições e critérios de aceite mensuráveis. ([OpenAI Developers][1])

## Cada ticket deve ter sempre:

* **Objetivo**
* **Contexto**
* **Escopo**
* **Fora de escopo**
* **Arquivos/áreas prováveis**
* **Critérios de aceite**
* **Comandos para validar**
* **Riscos de segurança**
* **Saída esperada do PR**

## Formato ideal para o Codex

Use sempre algo nesta linha:

```md
Objetivo:
Implementar X sem alterar Y.

Contexto:
Este projeto é zero-knowledge. O servidor não pode acessar segredos brutos.

Escopo:
- Fazer A
- Fazer B
- Fazer C

Fora de escopo:
- Não implementar D
- Não refatorar E

Arquivos prováveis:
- apps/web/src/...
- packages/crypto/src/...
- workers/api/src/...

Critérios de aceite:
- Cenário 1 funciona
- Cenário 2 falha corretamente
- Testes automatizados passam

Validação:
- pnpm test
- pnpm --filter web test
- pnpm lint

Entrega:
- código
- testes
- docs curtas
```

Isso reduz a chance de o agente sair “melhorando” metade do projeto porque viu uma oportunidade brilhante. Agentes adoram isso. Humanos também, infelizmente.

# 3. `AGENTS.md` recomendado para o repositório

Como o Codex lê `AGENTS.md` antes de trabalhar, vale começar o projeto já com esse arquivo. A própria OpenAI recomenda usar `AGENTS.md` para codificar padrões e expectativas do repositório. ([OpenAI Developers][2])

## Exemplo de `AGENTS.md`

```md
# AGENTS.md

## Projeto
Este projeto é um password manager self-hosted, zero-knowledge, com web app em Vue 3, extensão de navegador e backend em Cloudflare Workers.

## Princípios de segurança
- Nunca mover criptografia sensível para o servidor.
- Nunca enviar master password em texto claro.
- Nunca criar mecanismo de recuperação de cofre pelo admin.
- Account Kit não pode resetar master password.
- Todo anexo deve ser criptografado no cliente antes do upload.

## Stack
- Monorepo com pnpm
- Vue 3 + TypeScript no web app
- Browser extension separada
- Cloudflare Workers no backend
- D1 para metadados
- R2 para blobs criptografados

## Regras de implementação
- Prefira mudanças pequenas e localizadas.
- Não refatore arquivos não relacionados sem necessidade.
- Escreva testes para comportamento novo.
- Atualize tipos compartilhados quando alterar contratos.
- Documente decisões curtas em arquivos README locais quando necessário.

## Regras de UX
- Fluxos sensíveis devem ter mensagens claras e diretas.
- Sempre avisar que não existe recuperação de master password.
- Evitar atrito desnecessário em dispositivos já confiáveis.

## Regras de PR
Toda entrega deve incluir:
- resumo do que foi feito
- riscos ou limitações
- testes adicionados/ajustados
- itens pendentes claramente listados
```

# 4. PRD enxuto

## Nome provisório

**VaultLite**
Nome ruim, provisório, funcional. Como quase tudo em software antes do marketing estragar.

## Problema

Usuários que querem um gerenciador de senhas self-hosted, simples e seguro, com documentos pessoais e anexos criptografados, sem depender de recuperação por e-mail e sem confiar no servidor para acessar seus dados.

## Público inicial

* usuário individual avançado
* pequenos grupos de amigos/família no futuro
* self-hosters
* pessoas que querem guardar credenciais e documentos pessoais

## Proposta de valor

* simples de instalar
* zero-knowledge real
* bom no navegador
* suporta documentos e anexos desde o início
* sem falsa promessa de recuperação mágica

## Requisitos funcionais do v1

* onboarding por invite token
* criação de master password pelo usuário
* geração de Account Key
* geração de Account Kit
* login em novo dispositivo com username + master password + Account Key
* login item
* document item
* anexos criptografados
* sync do cofre
* busca e tags
* import/export
* troca de master password com sessão autenticada
* revogação básica de dispositivos

## Não objetivos do v1

* reset de senha por e-mail
* recuperação de cofre pelo admin
* compartilhamento complexo
* passkeys
* mobile nativo
* OCR
* thumbnails
* auditoria enterprise

## Sprint 1 - Biblioteca de cripto e modelo de conta

**Critérios de aceite**

**Validação**

```bash

```

**Instrução para Codex**

```md
Não invente esquema criptográfico próprio.
Use APIs/bibliotecas estáveis já escolhidas no projeto.
Mantenha a interface pequena e testável.
Não misture lógica de UI com lógica criptográfica.
```

# 12. Recomendação final de gestão do backlog

## Regras práticas

* cada ticket deve caber idealmente em **1 PR**
* cada PR deve ter **um objetivo único**
* tarefas grandes devem começar com **plan mode**
* toda feature sensível deve ter **teste**
* toda decisão de segurança deve ir para `SECURITY.md`
